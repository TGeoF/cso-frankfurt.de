# Repository Guidelines

## Project Structure & Module Organization
- `src/pages/` contains Astro page routes (lowercase, German slugs like `src/pages/konzerte.astro`).
- `src/components/` holds reusable UI pieces in PascalCase (e.g., `src/components/Navbar.astro`).
- `src/layouts/` defines shared page shells and wrappers.
- `src/styles/` stores global stylesheets (e.g., `src/styles/style.css`).
- `src/posts/` hosts Markdown content (currently `src/posts/text_leitung.md`).
- `public/` is for static assets copied to the build output.
- `dist/` is the build output (generated by `astro build`).

## Build, Test, and Development Commands
- `pnpm install` installs dependencies (lockfile is `pnpm-lock.yaml`).
- `pnpm dev` runs the local dev server at the default Astro port.
- `pnpm build` builds the static site into `dist/`.
- `pnpm preview` serves the production build for verification.

## Coding Style & Naming Conventions
- Use existing file naming: pages are lowercase; components are PascalCase.
- Follow Astro + TypeScript defaults; keep formatting consistent with current files.
- Prettier is installed; run `pnpm prettier --write .` if you add a config later.
- Prefer descriptive CSS class names that match the existing layout patterns in `src/styles/`.

## Testing Guidelines
- No automated test suite is configured.
- Do a manual smoke test: start `pnpm dev`, click through pages, and check layout regressions.

## Commit & Pull Request Guidelines
- Git history shows short, descriptive messages in German/English without a strict convention.
- Use concise commit subjects that describe the user-facing change.
- PRs should include: a short description, screenshots for UI changes, and linked issues if any.

## Configuration Tips
- A local `.env` file exists for environment-specific settings; do not commit secrets.
